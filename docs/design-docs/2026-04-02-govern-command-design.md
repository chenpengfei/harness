# /govern 命令设计文档

**日期**：2026-04-02
**状态**：已批准

---

## 概述

`/govern` 命令是 harness 仓库自身的持续治理工具。它通过三轮静默扫描（术语一致性、结构完整性、原则遵守）检测仓库健康状态，汇总为简报后逐项与用户协商处理，防止设计偏移和知识断层积累成"熵增"。

该命令仅适用于 harness 仓库本身，不安装到目标项目。

---

## 设计目标

- **对抗熵增**：定期识别术语漂移、结构缺口、原则违反，阻止细小偏差积累
- **人类掌舵**：每个问题均暂停询问用户，不自动批量修改
- **可自动修复优先**：可以机械替换的问题（术语）优先提供一键修复，复杂决策留给用户
- **无副作用扫描**：扫描阶段不修改任何文件，只收集问题列表

## 架构：三轮扫描 + 逐项协商

```
/govern
    │
    ▼
环境检查（INSTALL.md + ARCHITECTURE.md 存在性）
    │
    ▼
第一轮：术语一致性
  ├── E-K-C-F 框架名称
  ├── 四个维度中文名称（E=环境、K=知识、C=约束、F=反馈）
  ├── 人类掌舵表述
  ├── Agent 执行表述
  └── 安装后依赖表述
    │
    ▼
第二轮：结构完整性
  ├── Phase 文件完整性（8 个）
  ├── INSTALL.md 引用完整性
  ├── 设计文档索引覆盖
  ├── 命令-设计-执行闭环
  └── E-K-C-F 知识沉淀目录
    │
    ▼
第三轮：原则遵守
  ├── 原则 1：INSTALL.md 是单一真相来源
  ├── 原则 2：文档使用符号名称而非硬链接
  ├── 原则 3：do.md 保留完整人类审批检查点
  └── 原则 4：设计变更有对应记录
    │
    ▼
简报输出（统计数字）
    │
    ▼
逐项协商（优先级：原则违反 → 结构缺失 → 术语不一致 → 需人工确认）
    │
    ▼
协商完成摘要
```

## 关键设计决策

### 决策 1：扫描阶段静默收集，不逐条输出

**原则**：避免"扫描噪声"打断用户注意力，先完整收集再呈现。

扫描期间只显示进度（`正在扫描…（第 N/3 轮）`），所有问题收集为内部列表，扫描完成后一次性输出简报。

### 决策 2：可自动修复 vs 不可自动修复

两类问题的处理路径不同：

| 类型 | 选项 |
|------|------|
| 可自动修复（术语替换） | 立即修复 / 跳过 / 标记偏差 / 停止 |
| 不可自动修复（架构决策） | 我来处理 / 跳过 / 标记偏差 / 停止 |

纯术语替换可用 Edit 工具执行；凡涉及设计判断或文件创建的，均不可自动修复，须人工介入。

### 决策 3：不检查"命令文件不依赖外部 URL"

`/harness` 命令通过 WebFetch 从用户配置的 `harnessRepo` 拉取最新命令文件，属于运行时可选功能，URL 由用户配置决定，不是硬编码依赖。该约束过于严格且与 `/harness` 的设计意图冲突，故从原则列表中移除。

### 决策 4：协商优先级排序

原则违反 > 结构缺失 > 术语不一致 > 需人工确认。

原则违反影响最大（破坏架构不变式），结构缺失次之（知识断层），术语不一致属于表层问题，人工确认类不确定性最高放最后。

## 范围限制

- 仅适用于 harness 仓库本身（通过 INSTALL.md + ARCHITECTURE.md 存在性验证）
- 不检查目标项目安装后的健康状态（目标项目用 `/harness` 命令）
- 每次运行独立，不保留历史记录
