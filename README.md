# Harness

Agent-First 时代的工程支架。决定效率的不再是模型多强，而是团队能否[设计让 Agent 稳定产出的系统](https://openai.com/zh-Hans-CN/index/harness-engineering/)。

人类掌舵，智能体执行。软件工程团队的主要工作不再是编写代码，而是设计环境、明确意图和构建反馈回路，从而让 Agent 能够可靠地工作。

## 判断标准

- E-环境：让 Agent 能运行、调试、测试、观察系统
- K-知识：文档、规则、设计决策在代码仓库内可检索可维护
- C-约束：Lint、结构测试、依赖规则约束架构演化
- F-回路：日志、指标、评审、失败直接反馈给 Agent

## 极致实验 + 角色转变

- 从空代码仓库开始，不手写代码
- 由 Agent 生成产品代码、测试、CI、文档、工具
- 在真实使用中持续迭代
- 结论：Agent 当执行者 -> 工程体系必须重构

> 工程师角色重新定义
> - 从直接写代码 -> 设计让 Agent 稳定产出的系统
> - 把需求转成清晰的任务和验收标准
> - 把知识沉淀进仓库，不留在聊天和人脑里
> - 把规范做成可执行规则
> - 让 Agent 能直接观察、验证、修复问题

我们并不认为 AI 会完全替代工程师，而是认为：如果 Agent 是主要执行者，工程体系本身必须重构。


## 三个最值得重视的启发

- 知识是否“在仓库里”变得非常关键

    Agent 运行时读不到的知识 = 不存在

    文档不再是辅助材料，而是生产系统的一部分

- 严格边界比过去更重要

    Agent 产出速度快，没有清晰的架构层次、依赖方向、机械化规则，系统会更快失控而不是更快进步

- 测试与可观测性 = Agent 的工作输入

    不只是给人看质量工具，也是训练 Agent 自我修正的反馈机制

    日志、指标、评审正在变成 Agent 的眼镜和耳朵


Agent-First 时代的工程支架 = 环境 x 知识 x 约束 x 回路

模型能力是地基，但真正让 Agent 稳定产出的是围绕它构建的工程体系，如何让整个仓库对 Agent 友好是关键

---

该工程的目的是将以上 Harness Engineering 理念赋能到你的代码仓库中。

# 安装指南

将以下指令复制给你的 Agent（Claude Code 或其他支持 WebFetch 的 Agent）：

```
请读取 https://raw.githubusercontent.com/<your-username>/harness/main/INSTALL.md
并按照其中的指引，在当前项目目录下安装 Harness 能力。
```

Agent 会自动完成安装，过程中会逐步向你确认每个阶段。

## 更新已安装的 Harness

安装完成后，在 Claude Code 中运行：

```
/harness
```

该命令会重新评估你的项目配置，并根据变化更新相应的文档和配置。

若需了解安装详情，可直接查阅 [INSTALL.md](./INSTALL.md)。
