# Harness 反馈机制

本文档描述 harness 的反馈回路：改进如何从实践中流回 harness 仓库。

---

## 反馈层级

| 层级 | 触发者 | 机制 |
|------|--------|------|
| 开发反馈 | harness 贡献者 | `/govern` 命令持续治理，扫描仓库健康状态并逐项协商处理 |
| 使用反馈 | 目标项目用户 | 直接在 harness 仓库提交 GitHub Issue |

---

## `/govern` 治理回路

在 harness 仓库开发过程中，`/govern` 命令作为持续治理回路：

- 四轮扫描：术语一致性、结构完整性、原则遵守、Agent 友好性
- 逐项暂停，与用户协商每个发现；可选自动修复或跳过
- 每次 `feat:` 或重大修改后建议运行，对抗熵增

> **分工说明**：`/govern` 负责全局治理（含一致性审查）；若只需快速核查特定一致性问题，可单独运行 `/audit`。

---

## 改进建议处理流程

```
目标项目实践发现改进点 → 在 harness 仓库提交 GitHub Issue
                                        ↓
                          harness 维护者评审 / 讨论
                                        ↓
                          docs/design-docs/ 设计更新
                                        ↓
                          install/phase-*.md 实现
```

---

## /govern 检查点

每次发布前建议通过 `/govern` 零问题状态：

```
✓ 本轮治理完成，未发现需处理问题，当前 harness 仓库状态健康。
```
