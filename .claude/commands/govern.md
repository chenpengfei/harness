## 环境检查

用 Glob 检查根目录是否同时存在 `INSTALL.md` 和 `ARCHITECTURE.md`。

若两者不同时存在，停止执行并输出：

    错误：当前目录不是 harness 仓库。
    /govern 命令仅适用于 harness 仓库本身，请在 harness 仓库根目录下运行。

## 扫描阶段

**静默执行以下三轮检查，将所有发现收集为内部列表，不逐条输出。**

扫描时在终端显示进度：`正在扫描…（第 N/3 轮）`

收集结果时，每条记录包含以下字段：
- `轮次`：第一轮 / 第二轮 / 第三轮
- `类别`：术语不一致 / 结构缺失 / 原则违反 / 需人工确认
- `位置`：文件名:行号（可知则填，否则留空）
- `描述`：具体问题描述
- `可自动修复`：是 / 否

---

### 第一轮：术语一致性

用 Glob 列出全仓库所有 `.md` 文件（`**/*.md`，排除 `.git/` 目录），逐一读取，检查以下五组术语：

**术语组 1 — E-K-C-F 框架名称**
标准写法：`E-K-C-F`
收录：`EKCF`、`E/K/C/F`、`ekc-f` 等不同写法同时出现在仓库中

**术语组 2 — 四个维度中文名称**
标准对应：E=环境、K=知识、C=约束、F=反馈
收录：将 F 对应"回路"而非"反馈"，或将 C 对应"限制"而非"约束"

**术语组 3 — 人类主导表述**
标准写法：`人类掌舵`
收录：`人类控制`、`用户控制`、`人工介入`（类别标为"需人工确认"）

**术语组 4 — Agent 执行表述**
标准写法：`Agent 执行`（`智能体执行` 在 README 标语中为合法变体，不收录）
收录：`AI 执行`、`自动执行`

**术语组 5 — 安装后依赖表述**
标准写法：`安装后无依赖` 或 `安装后无运行时依赖`
用 Grep 搜索 `无运行时依赖`、`安装后脱离依赖` 等变体，若与标准写法在不同文件中混用则收录

---

### 第二轮：结构完整性

**2.1 Phase 文件完整性**

检查以下 8 个文件是否存在，缺失则收录（类别：结构缺失，不可自动修复）：

- `install/phase-0-preflight.md`
- `install/phase-1-info-collection.md`
- `install/phase-2-environment.md`
- `install/phase-3-knowledge.md`
- `install/phase-4-constraints.md`
- `install/phase-5-feedback.md`
- `install/phase-6-finalize.md`
- `install/phase-7-agent-team.md`

**2.2 INSTALL.md 引用完整性**

读取 `INSTALL.md`，用 Grep 提取所有 `install/phase-[0-9]+-[a-z-]+\.md` 路径引用，去重后检查每个文件是否存在。"引用存在但磁盘没有"的收录为：引用断裂，不可自动修复。

**2.3 设计文档索引**

检查 `docs/design-docs/index.md` 是否存在；若存在，检查其是否覆盖 `docs/design-docs/` 下每个 `.md` 文件（除自身）。缺失则收录。

**2.4 命令-设计-执行闭环**

枚举 `.claude/commands/` 下所有命令文件，对每个 `<name>.md`：
- 检查 `docs/design-docs/*<name>*` 是否存在
- 检查 `docs/exec-plans/completed/*<name>*` 是否存在

缺失则分别收录为：缺少设计文档 / 缺少完成的执行计划（均不可自动修复）。

**2.5 E-K-C-F 知识沉淀**

检查以下目录是否存在且含至少一份 `.md` 文件：
`.harness/environment/`、`.harness/knowledge/`、`.harness/constraints/`、`.harness/feedback/`

缺失则收录（不可自动修复）。

---

### 第三轮：原则遵守

**原则 1：INSTALL.md 是单一真相来源**

比对 `INSTALL.md` 中各 Phase 描述与对应 `phase-*.md` 标题。明显矛盾的收录为：违反原则 1；无法机械判断的收录为：需人工确认。

**原则 2：文档使用符号名称而非硬链接**

Grep 搜索所有 `.md` 文件中 `](../`、`](./`、`](/` 且目标以 `.md)` 结尾的内部链接。
若找到，收录为：违反原则 2，可自动修复（删除链接，改为符号名称引用需人工判断，故标为不可自动修复）。

**原则 3：do.md 保留完整人类审批检查点**

读取 `.claude/commands/do.md`，检查三个检查点特征词：
- `设计方案审批` 或 `批准，开始实现`
- `实现已完成` 或 `继续全部`
- `PR 合并审批` 或 `批准合并`

任一检查点两个关键词均不存在，收录为：违反原则 3，不可自动修复。

**原则 4：设计变更有对应记录**

执行 `git log --pretty="%ad %s" --date=short -20`，获取最近 20 条提交。
对含 `feat:` 或 `refactor:` 的提交，检查 `docs/design-docs/` 中是否存在日期 ≤ 提交日期的设计文档。
无对应文档的，收录为：需人工确认（是否需要补充设计文档）。

## 简报输出

扫描完成后，输出简报（仅统计数字，不展示详情）：

```
────────────────────────────────
治理简报
────────────────────────────────
第一轮  术语一致性      N 个问题
第二轮  结构完整性      N 个问题
第三轮  原则遵守        N 个问题（其中 N 项需人工确认）
────────────────────────────────
合计：X 个问题，其中 Y 项可自动修复
────────────────────────────────
```

若所有问题数均为 0，输出：

```
✓ 仓库状态健康，未发现问题。
```

然后结束执行。

## 逐项协商

按以下顺序依次处理每个问题（优先级：原则违反 → 结构缺失 → 术语不一致 → 需人工确认）：

对每个问题，输出：

```
问题 M/X
────────────────────────────────
[类别] 轮次 · 位置
描述：……
影响：……（简要说明为什么这会造成熵增或设计偏移）
可自动修复：是 / 否
────────────────────────────────
```

然后用 AskUserQuestion 暂停，选项根据"可自动修复"字段动态生成：

**可自动修复时：**
- 立即修复（执行修复后继续下一项）
- 跳过（本次不处理）
- 标记为已知偏差（记录到本次会话摘要，不再重复提示）
- 停止协商（结束逐项流程，输出当前进度摘要）

**不可自动修复时：**
- 我来处理（记录用户已知晓，继续下一项）
- 跳过（本次不处理）
- 标记为已知偏差（记录到本次会话摘要，不再重复提示）
- 停止协商（结束逐项流程，输出当前进度摘要）

选择"立即修复"时，执行修复（仅术语替换类可用 Edit 工具），完成后输出：`✓ 已修复`，然后继续下一项。

若修复失败（匹配不到目标文本），输出失败原因，将该条改为"需人工处理"后继续。

---

## 协商完成摘要

全部问题协商完毕后（或用户选择"停止协商"后），输出：

```
────────────────────────────────
本次治理摘要
────────────────────────────────
已修复：N 项
已处理（用户接手）：N 项
跳过：N 项
标记为已知偏差：N 项
────────────────────────────────
剩余待处理：N 项
```

若存在剩余问题，附上待处理清单（文件名:行号 + 一行描述）供用户参考。
