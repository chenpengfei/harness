## 入口检查

用 Glob 检查是否存在 `.harness/` 目录。

若不存在，停止执行并输出：

    错误：当前工程未安装 Harness（未找到 .harness/ 目录）。
    /govern 命令适用于已安装 Harness 的工程，请在正确的目录下运行。

---

## 扫描阶段

**静默执行以下四轮检查，将所有发现收集为内部列表，不逐条输出。**

扫描时在终端显示进度：`正在扫描…（第 N/4 轮）`

收集结果时，每条记录包含以下字段：
- `轮次`：第一轮 / 第二轮 / 第三轮 / 第四轮
- `类别`：结构缺失 / 文档不一致 / 合约缺失 / 模式不统一 / 需人工确认
- `位置`：文件名:行号（可知则填，否则留空）
- `描述`：具体问题描述
- `可自动修复`：是 / 否

---

### 第一轮：结构完整性
*对应 Agent 友好性原则 3 — 防止错误复合*

**1.1 E-K-C-F 知识沉淀目录**

检查以下目录是否存在且含至少一份 `.md` 文件：
`.harness/environment/`、`.harness/knowledge/`、`.harness/constraints/`、`.harness/feedback/`

缺失则收录（类别：结构缺失，不可自动修复）。

**1.2 Harness 配置文件**

若 `.harness/harness-config.json` 存在，检查是否包含 `project.name` 和 `techStack` 字段（非空）。缺失字段收录为：结构缺失，不可自动修复。

**1.3 CLAUDE.md 存在性**

检查 `CLAUDE.md` 是否存在。缺失则收录（类别：结构缺失，不可自动修复）。

---

### 第二轮：文档与现实一致
*对应 Agent 友好性原则 2 — 文档与代码一致*

**2.1 CLAUDE.md 健康度**

读取 `CLAUDE.md`（若存在）：
- 若行数超过 200 行，收录为：文档不一致（CLAUDE.md 过长，应保持 200 行以内），不可自动修复
- 若含未填写占位符（`<xxx>`、`[待填写]`、`TODO:`），收录为：文档不一致（含未填写占位符），不可自动修复

**2.2 knowledge/architecture.md 实质性**

若 `.harness/knowledge/architecture.md` 存在，检查行数是否 > 15。若 ≤ 15 行，收录为：文档不一致（架构文档未实质填写），不可自动修复。

**2.3 无内部硬链接**

Grep 搜索所有 `.md` 文件中 `](../`、`](./`、`](/` 且目标以 `.md)` 结尾的内部链接。若找到，收录为：文档不一致（使用硬链接而非符号名称引用），不可自动修复。

---

### 第三轮：合约与知识沉淀
*对应 Agent 友好性原则 1 — 测试即合约*

**3.1 do.md 工作流合约完整性**

若 `.claude/commands/do.md` 存在，检查三个检查点特征词：
- `设计方案审批` 或 `批准，开始实现`
- `实现已完成` 或 `继续全部`
- `PR 合并审批` 或 `批准合并`

任一检查点两个关键词均不存在，收录为：合约缺失（do.md 丢失人类审批检查点），不可自动修复。

**3.2 ADR 知识沉淀**

若 `.harness/knowledge/decisions/` 目录存在，检查其是否含至少 1 个 `.md` 文件。若为空目录，收录为：合约缺失（决策目录存在但无任何 ADR），不可自动修复。

**3.3 设计先于实现**

执行 `git log --pretty="%ad %s" --date=short -20`，获取最近 20 条提交。
对含 `feat:` 或 `refactor:` 的提交，检查是否存在日期 ≤ 提交日期的 ADR（`.harness/knowledge/decisions/` 下），或设计文档（`docs/design-docs/` 下，若该目录存在）。
无对应文档的，收录为：需人工确认（是否需要补充设计文档或 ADR）。

> 若 `git log` 执行失败（非 git 环境），跳过本检查项，记为"已跳过（非 git 环境）"。

---

### 第四轮：设计模式统一
*对应 Agent 友好性原则 4 — 设计模式统一*

用 Glob 列出全仓库所有 `.md` 文件（`**/*.md`），逐一读取，检查以下术语组：

**术语组 1 — E-K-C-F 框架名称**
标准写法：`E-K-C-F`
收录：`EKCF`、`E/K/C/F`、`ekc-f` 等不同写法同时出现在仓库中

**术语组 2 — 四个维度中文名称**
标准对应：E=环境、K=知识、C=约束、F=反馈
收录：将 F 对应"回路"而非"反馈"，或将 C 对应"限制"而非"约束"

**术语组 3 — Agent 执行表述**
标准写法：`Agent 执行`（`智能体执行` 在 README 标语中为合法变体，不收录）
收录：`AI 执行`、`自动执行`

**术语组 4 — 安装后依赖表述**
标准写法：`安装后无依赖` 或 `安装后无运行时依赖`
用 Grep 搜索 `无运行时依赖`、`安装后脱离依赖` 等变体，若与标准写法在不同文件中混用则收录

**术语组 5 — 旧命令名称引用**
用 Grep 搜索所有 `.md` 文件中出现 `/audit` 的位置。若在当前功能描述中用 `/audit` 指代治理功能，收录为：模式不统一（应使用 `/govern`），可自动修复。

## 简报输出

扫描完成后，输出简报（仅统计数字，不展示详情）：

```
────────────────────────────────
治理简报
────────────────────────────────
第一轮  结构完整性（防止错误复合）    N 个问题
第二轮  文档与现实一致                N 个问题
第三轮  合约与知识沉淀                N 个问题（其中 N 项需人工确认）
第四轮  设计模式统一                  N 个问题
────────────────────────────────
合计：X 个问题，其中 Y 项可自动修复
────────────────────────────────
```

若所有问题数均为 0，输出：

```
✓ 仓库状态健康，未发现问题。
```

然后结束执行。

## 逐项协商

按以下顺序依次处理每个问题（优先级：合约缺失 → 结构缺失 → 文档不一致 → 模式不统一 → 需人工确认）：

对每个问题，输出：

```
问题 M/X
────────────────────────────────
[类别] 第N轮 · 位置
描述：……
影响：……（简要说明为什么这会造成熵增或设计偏移）
可自动修复：是 / 否
────────────────────────────────
```

然后用 AskUserQuestion 暂停，选项根据"可自动修复"字段动态生成：

**可自动修复时：**
- 立即修复（执行修复后继续下一项）
- 跳过（本次不处理）
- 标记为已知偏差（记录到本次会话摘要，不再重复提示）
- 停止协商（结束逐项流程，输出当前进度摘要）

**不可自动修复时：**
- 我来处理（记录用户已知晓，继续下一项）
- 跳过（本次不处理）
- 标记为已知偏差（记录到本次会话摘要，不再重复提示）
- 停止协商（结束逐项流程，输出当前进度摘要）

选择"立即修复"时，执行修复（仅术语替换类可用 Edit 工具），完成后输出：`✓ 已修复`，然后继续下一项。

若修复失败（匹配不到目标文本），输出失败原因，将该条改为"需人工处理"后继续。

---

## 协商完成摘要

全部问题协商完毕后（或用户选择"停止协商"后），输出：

```
────────────────────────────────
本次治理摘要
────────────────────────────────
已修复：N 项
已处理（用户接手）：N 项
跳过：N 项
标记为已知偏差：N 项
────────────────────────────────
剩余待处理：N 项
```

若存在剩余问题，附上待处理清单（文件名:行号 + 一行描述）供用户参考。
